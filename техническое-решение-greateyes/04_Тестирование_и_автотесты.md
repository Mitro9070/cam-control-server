# Тестирование и автотесты

## 1. Цели тестирования
- Подтвердить корректность сквозного сценария от ASCOM до SDK.
- Гарантировать стабильность состояния камеры при долгой работе.
- Раннее выявление регрессий через автотесты на каждом слое.

## 2. Уровни тестирования
## 2.1 Unit tests
### C# ASCOM Local Server
- Маппинг API-ошибок в ASCOM исключения.
- Валидация параметров (`Duration`, `BinX/BinY`, ROI).
- Логика polling `ImageReady`.

### Python service
- State machine камеры (`disconnected -> idle -> exposing -> reading -> idle/error`).
- Валидация контрактов endpoint'ов.
- Нормализация SDK ошибок.
- Репозитории БД (CRUD профилей/настроек/логов).

### SDK adapter
- Типы/маршалинг входных/выходных параметров.
- Поведение таймаутов и retry.
- Корректность преобразования буфера изображения.

## 2.2 Integration tests
- Python API + PostgreSQL (testcontainer).
- Python API + mock SDK DLL.
- C# Local Server + Python API (контрактные тесты).

## 2.3 End-to-end tests
- Сквозной сценарий:
  `ASCOM client simulator -> Local Server -> Python -> SDK mock -> image result`.
- UI smoke e2e:
  connect, start exposure, status polling, image ready, logs visible.

## 3. Матрица кейсов
## 3.1 Подключение
- `TC-CONN-001`: успешный connect.
- `TC-CONN-002`: connect при отключенной DLL.
- `TC-CONN-003`: повторный connect без disconnect.
- `TC-CONN-004`: disconnect из каждого допустимого состояния.

## 3.2 Экспозиция
- `TC-EXP-001`: успешная light-экспозиция.
- `TC-EXP-002`: dark-экспозиция (shutter closed).
- `TC-EXP-003`: abort во время exposing.
- `TC-EXP-004`: stop во время reading.
- `TC-EXP-005`: таймаут SDK при чтении кадра.
- `TC-EXP-006`: серия из N экспозиций без деградации.

## 3.3 Параметры камеры
- `TC-CFG-001`: валидный биннинг.
- `TC-CFG-002`: невалидный биннинг (ожидаем error).
- `TC-CFG-003`: валидный ROI.
- `TC-CFG-004`: выход ROI за границы сенсора.
- `TC-CFG-005`: корректная выдача `CameraXSize/CameraYSize`.
- `TC-CFG-006`: корректная выдача `SensorName` и `HasShutter`.

## 3.4 Cooling
- `TC-CL-001`: включение cooler.
- `TC-CL-002`: установка target температуры.
- `TC-CL-003`: превышение backside temp -> error/auto-off.
- `TC-CL-004`: `CoolerPower` (если поддержан SDK) возвращает валидный диапазон 0..100.
- `TC-CL-005`: warm-up сценарий выполняется ступенчато, без резкого `cooler_off`.

## 3.5 БД
- `TC-DB-001`: миграции применяются с нуля.
- `TC-DB-002`: миграции обратимы (downgrade/upgrade).
- `TC-DB-003`: сохранение профилей и настроек.
- `TC-DB-004`: запись событий с correlation id.
- `TC-DB-005`: режим R1 без PostgreSQL работает штатно (fallback storage).

## 3.6 UI
- `TC-UI-001`: connect/disconnect.
- `TC-UI-002`: старт/остановка экспозиции.
- `TC-UI-003`: отображение статусов и ошибок.
- `TC-UI-004`: редактирование и сохранение профиля.

## 4. Автотесты "сразу"
## 4.1 Обязательный набор в первой итерации
- Python:
  - `pytest` unit + integration.
  - Контрактные тесты API (schemas, status codes).
  - testcontainer PostgreSQL.
- C#:
  - `xUnit`/`NUnit` unit tests для слоя адаптера.
  - Моки HTTP клиента к Python API.
- UI:
  - `Playwright` smoke e2e на ключевые сценарии.

## 4.2 CI-gates (минимум)
- `lint + unit` на каждый PR.
- `integration` на merge в main/dev.
- `e2e smoke` nightly и перед релизом.

## 5. Критерии приемки
- Все обязательные тесты зеленые.
- No critical regression по матрице кейсов.
- Успешный сквозной тест с реальной камерой на стенде.
- Успешный прогон Conform/ConformU на целевом наборе функций.

## 6. Тестовые данные и окружения
- `dev`: mock SDK + локальный storage (без обязательной БД).
- `dev-db`: mock SDK + PostgreSQL.
- `staging`: реальный SDK + тестовая камера (при наличии).
- `prod`: реальные камеры, с ограниченным доступом к test endpoints.

## 7. Отчетность
- Автогенерация junit/xml отчётов.
- Покрытие кода (минимум 70% для Python service core).
- Отдельный runbook по ручной валидации на железе.
