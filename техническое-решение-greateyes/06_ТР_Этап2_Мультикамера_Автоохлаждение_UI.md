## 5. Автоохлаждение: принципы и выбранный алгоритм

## 5.1 Что важно для долгих наблюдений
- Критично держать **стабильный setpoint** для совместимости light/dark.
- Избегать длительного насыщения TEC по мощности.
- Нагрев (warmup) делать ступенчато, чтобы не перегружать камеру.
- Следить за backside temperature и реагировать на аварийные статусы.

## 5.2 Режимы управления
- `Safe` - минимальный риск, медленный ramp.
- `Balanced` - режим по умолчанию.
- `Fast` - ускоренный выход на setpoint, затем авто-переход в Balanced.

## 5.3 Алгоритм
Выбран гибрид:
- планировщик setpoint по ramp;
- PI-контур с anti-windup;
- защита по лимитам температуры и по backside;
- опциональный учет внешней температуры/влажности (если введены вручную или доступны из внешнего источника).

Псевдологика:
1. Принять `target_temp_c` и профиль (`Safe/Balanced/Fast`).
2. Построить `target_plan` с ограничением скорости изменения.
3. Каждую секунду считать `sensor_temp` и `backside_temp`.
4. Рассчитать мощность PI-контуром с anti-windup.
5. Ограничить мощность диапазоном безопасности.
6. Если `backside_temp` близко к пределу или SDK дает тревожный статус:
   - снять мощность,
   - перевести в safe mode,
   - поднять событие в лог и UI.

## 5.4 Warmup и MaxIm
- В ASCOM нет отдельной стандартной команды `WarmUp`.
- На этапе 2 ввести правило:
  - `CoolerOn=false` и/или `SetCCDTemperature` к более теплому target запускает управляемый warmup-процесс.
- Добавить debug-трассировку вызовов из MaxIm для точного маппинга поведения кнопки warmup.

## 3. Best practices для долговременных наблюдений (основа проектирования)

1. Главное для качества калибровки: **стабильная температура сенсора**, а не «минимально возможная любой ценой».
2. Нужна **одинаковая температура для light/dark** кадров и повторяемые режимы gain/readout.
3. Охлаждение и особенно нагрев выполняются **плавно** (ramp), без резких переходов.
4. Не допускать режима, где TEC работает в постоянной насыщенной зоне (близко к 100% долгое время).
5. Требуется контроль риска конденсации через запас к точке росы (если доступны погодные данные).

## 4. Требования к автоохлаждению

## 4.1 Функциональные
- Пользователь задает `target_temp_c` (или выбирает профиль), но **не управляет вручную мощностью кулера**.
- Система автоматически регулирует `cooler_power_percent`.
- Поддержка профилей:
  - `Safe` (минимум риска, медленнее),
  - `Balanced` (по умолчанию),
  - `Fast` (быстрый выход на setpoint).
- Поддержка режимов источника окружающей температуры:
  - `manual_ambient` (ввод вручную),
  - `weather_auto` (по геолокации и погодному API),
  - `sensor_auto` (если в будущем появится датчик среды).

## 4.2 Нефункциональные
- Выход на setpoint (Balanced): до 15 минут для шага 20C (зависит от среды).
- Перерегулирование: не более 1.0C (Balanced), не более 2.0C (Fast).
- Удержание: не хуже +/-0.3C в стационаре.
- Безопасность: при приближении к точке росы вводить защитное ограничение setpoint.

## 5. Предлагаемый оптимальный алгоритм (выбранный)

## 5.1 Почему не один «жесткий PID»
Чистый PID без ограничений неустойчив в полевых условиях из-за:
- насыщения по мощности TEC,
- изменения тепловой инерции камеры в зависимости от обдува/среды,
- ночных трендов температуры.

Поэтому выбирается **гибридный алгоритм**: ramp + PI с anti-windup + dew-point guard.

## 5.2 Алгоритм Hybrid Cooling Controller

### Шаг 1: План setpoint по ramp
- Формируем `target_plan(t)` с ограничением скорости:
  - cooling ramp: например `0.6 C/min` (Balanced),
  - warmup ramp: `0.8 C/min` (безопасный нагрев).
- Это исключает резкие переходы и уменьшает overshoot.

### Шаг 2: PI-контур с anti-windup
- Ошибка: `e = target_plan - ccd_temp`.
- Управление:
  - `u = Kp * e + I`,
  - `I` интегрируется только когда выход не в жестком насыщении или когда ошибка ведет к выходу из насыщения.
- Ограничения:
  - `u` в диапазоне `[0..u_max]`,
  - `u_max` динамический (обычно 85-90% для долгой стабильной работы).
- Deadband:
  - при `|e| < 0.2C` снижать активность интегратора для стабилизации.

### Шаг 3: Защита от конденсации
- При доступной внешней температуре/влажности считаем dew point.
- Вводим ограничение:
  - `target_floor = dew_point - margin`,
  - margin по профилю (`Safe` больше, `Fast` меньше).
- Если пользователь просит ниже безопасного предела:
  - статус `clamped_by_dew_point`,
  - лог и уведомление в UI.

### Шаг 4: Адаптация к среде
- Если `ambient` растет и TEC близок к насыщению:
  - плавно поднимать `target_plan` до достижимого диапазона;
  - сохранять предсказуемость dark frames через фиксированные ступени (например, -10/-5/0).

## 5.3 Псевдокод

```text
loop every 1s:
  read ccd_temp, cooler_power, ambient(optional), humidity(optional)
  target_safe = apply_dew_guard(user_target, ambient, humidity, profile)
  target_plan = ramp_to_target(prev_target_plan, target_safe, profile_ramp_rate)

  e = target_plan - ccd_temp
  u_p = Kp * e
  u_i = update_integrator_with_anti_windup(u_prev, e, limits, profile)
  u = clamp(u_p + u_i, 0, u_max(profile, ambient))

  if abs(e) < deadband: apply_hold_mode()
  set_cooler_power_percent(u)
  publish telemetry/status
```

## 6. Кастомные алгоритмы для выбора пользователем

Чтобы не усложнять интерфейс и одновременно дать гибкость:

1) **Safe (по умолчанию для влажной ночи)**
- медленный ramp;
- широкий запас к dew point;
- conservative `u_max`.

2) **Balanced (основной режим)**
- умеренный ramp;
- стандартный запас к dew point;
- оптимум «стабильность/скорость».

3) **Fast (быстрый старт сессии)**
- более агрессивный ramp и Kp;
- сниженный запас к dew point (но не ниже hard-limit);
- автоматический переход в Balanced после достижения setpoint.

## 7. Геолокация, погода и ручной ввод

## 7.1 Режимы источника среды
- `Manual`: оператор вводит `ambient_temp_c` и `humidity_percent`.
- `Geo+Weather`: система по координатам обсерватории читает погоду и обновляет ambient/dew point.
- `Hybrid`: если weather недоступен, fallback на последние валидные данные + ручной override.

## 7.2 Что выбрать как дефолт
- По умолчанию: `Manual + optional weather assist`.
- Причина: сетевые API могут быть недоступны ночью, а управление охлаждением должно оставаться автономным.

- При добавлении второй камеры автоматически отображается **вторая панель управления**:
  - независимые connect/disconnect;
  - независимые exposure controls;
  - независимые графики температуры и статусов.

## 10. Исследования по усилителям (пункты 46-49 замечаний)

## 10.1 Гипотеза
- 1 усилитель дает лучший контроль шума для спектральных задач,
- 2/4 усилителя ускоряют считывание, но могут повысить неравномерность/шум.

## 10.2 План эксперимента
- Параметры:
  - amplifiers: 1/2/4,
  - speed: 50kHz/250kHz/1MHz,
  - gain: HC/LN,
  - temp: -5/-10/-15C.
- Наборы:
  - bias: 100 кадров;
  - dark: 50 кадров на 60/300с;
  - light test pattern: 30 кадров.
- Метрики:
  - RMS noise, hot pixels, readout time, quadrant mismatch.

## 10.3 Критерии выбора дефолта
- Для "долгие спектры": минимальный шум и стабильность baseline.
- Для "быстрая фокусировка/поиск": минимальное время readout.
- Результат: таблица рекомендованных профилей.

## 13. Критерии приемки этапа 2

- 2 камеры управляются независимо в одном UI.
- У второй камеры автоматически появляется отдельная панель управления.
- Выбор порта/IP и сохранение профилей работает без ручного редактирования `.env`.
- Cooling auto-mode удерживает температуру в пределах SLA и не требует ручного процента.
- FITS export и metadata viewer работают на последнем кадре.
- Рекомендованный amplifier режим подтвержден измерениями и зафиксирован в документации.

## 14. Риски и смягчение

- **Риск:** нестабильная погода API -> **смягчение:** manual/hybrid fallback.
- **Риск:** перерегулирование TEC -> **смягчение:** anti-windup + ramp + profile tuning.
- **Риск:** рост сложности UI -> **смягчение:** поэтапный rollout и feature flags.
- **Риск:** различия между экземплярами камер -> **смягчение:** профили per camera_id + отдельная калибровка.

## 15. Вывод

Оптимальная стратегия этапа 2:
- внедрить гибридный автоалгоритм охлаждения (ramp + PI anti-windup + dew-point guard),
- добавить мультикамерную модель и современный операторский UI,
- формализовать выбор режима усилителей на основе измерений, а не предположений.

Это даст максимальную эксплуатационную пользу для долгих наблюдений и снизит риск деградации качества данных при реальных погодных колебаниях.
