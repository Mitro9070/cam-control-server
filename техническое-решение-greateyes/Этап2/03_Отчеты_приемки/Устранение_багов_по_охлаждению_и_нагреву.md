# Устранение багов по охлаждению и нагреву

Документ фиксирует исправления в Python-сервисе камеры (`02_Python_Camera_Service`), ASCOM-драйвере (`01_ASCOM_Local_Server`) и веб-интерфейсе (`05_UI`), а также описывает актуальную логику работы охлаждения, прогрева и синхронизации с клиентами (в т.ч. MaxIm DL).

**Версия логики сервиса:** `runtime_version` в ответе `GET /api/v1/camera/cooling/debug` (на момент документа — **2.4**).

## 1. Что было не так (кратко)

| Симптом | Причина |
|--------|---------|
| Цель температуры «прыгала» или сбрасывалась на текущую CCD через ~1–2 с | Контрольная петля отправляла в SDK старый «рамп» (`control_target`), либо при `cooler_off` цель перезаписывалась значением `ccd_temp_c`. |

| `ERR_CONNECTION_REFUSED` на 3037 | Процесс `uvicorn` был остановлен — на порту никто не слушал. |

## 2. Что исправлено в коде

### 2.1. Python: `camera_runtime.py`

- **Единый смысл переменных**
  - `_requested_target_temp_c` / `_state.target_temp_c` — **цель оператора** (что показываем в API и ASCOM).
  - `_control_target_temp_c` — **внутренняя** цель для рампа и вызовов SDK; при смене операторской цели **снапится** к ней, чтобы не отправлять в SDK устаревшее значение.

- **PI-контроллер мощности (индикатор)**  
  Ошибка: `ccd_temp - control_target` (положительная — CCD теплее цели → нужна большая «мощность» в нашей модели).

- **Повторный `connect()`**  
  Если камера уже подключена и SDK отвечает — **сохраняется** состояние охлаждения и цель (типичный сценарий: новый COM-объект MaxIm DL).

- **`set_target_temperature`**  
  Не включает кулер сам по себе; в SDK цель уходит **только если** `cooler_on == true`. Иначе цель только сохраняется до включения кулера.

- **`set_cooler_power(False)`**  
  Кулер выключается, но **цель оператора не затирается** текущей температурой CCD (иначе MaxIm/UI «теряли» setpoint).

- **`set_cooler_power(True)`**  
  Сразу отправляет в SDK сохранённую цель; при явной мощности сидируется интегральная составляющая PI.

- **Самовосстановление рассинхрона**  
  Если CCD и `control_target` оказываются по разные стороны от запрошенной цели — `control_target` ресинхронизируется с запросом.

- **Ошибки диапазона SDK**  
  При `apply_target_failed` с текстом `out of range A..B` выполняется попытка clamp и повторной установки.

- **Warm-up (`_run_warmup`)**  
  - Обработка выхода за диапазон SDK (clamp, корректировка финальной цели job при необходимости).  
  - Поток не падает «тихо»: ошибки переводят job в `error` с сообщением.  
  - **После завершения warm-up** кулер остаётся в режиме регулирования: цель фиксируется на достигнутой, восстанавливается режим контроллера (`balanced` и т.д.), включается closed-loop удержание (а не простое отключение с последующим «уездом» температуры).

### 2.2. Python: API (`routes.py`, `schemas.py`)

- Логирование вызовов: `PUT` мощности/цели, статус контроллера.  
- Ответ warm-up дополнен полями `target_temp_c` и `error` (где применимо).

### 2.3. ASCOM: `CameraDriver.cs`

- В `TraceLogger` пишутся операции **чтения/записи** `CoolerOn` и `SetCCDTemperature` (для разбора расхождений с MaxIm DL).

### 2.4. UI: `05_UI/web/app.js`

- Для warm-up числа с **запятой** (`0,5`) корректно парсятся (замена `,` на `.`).

### 2.5. Диагностика

- `GET /api/v1/camera/cooling/debug` — снимок состояния (`runtime_version`, цели, последний SDK-target, режим, warm-up).  
- В лог-файл пишутся строки `COOLING_TRACE` (тики контроллера и ключевые события).

**Путь лога:** `02_Python_Camera_Service/logs/camera_service.log`.

## 3. Как теперь работает (поведение для оператора)

### 3.1. Обычное охлаждение / удержание температуры

1. Включить охладитель (`cooler_on = true` — UI, API или ASCOM `CoolerOn`).  
2. Задать цель (`PUT /camera/cooling/target` или ASCOM `SetCCDTemperature`).  
3. Сервис хранит **цель оператора** отдельно от внутреннего рампа; PI обновляет отображаемую мощность; периодически в SDK уходит согласованная цель (с учётом debounce и диапазона SDK).

### 3.2. Выключение кулера

- Кулер выключается, **заданная цель в сервисе сохраняется** — при следующем включении кулера она снова уйдёт в SDK.

### 3.3. Плавный прогрев (Warm-up)

- Параметры: цель, шаг по температуре, шаг по мощности, интервал.  
- Если запрошенная цель **выше максимума SDK** (у вас часто **20 °C**), фактическая конечная цель **ограничивается** диапазоном — это ограничение прошивки/SDK, не баг прикладного уровня.  
- После достижения условий завершения warm-up включается **режим удержания** цели основным контроллером (без «отпускания» датчика вверх без регулирования).

### 3.4. MaxIm DL и ASCOM

- **Setpoint** в MaxIm соответствует тому, что возвращает драйвер при чтении `SetCCDTemperature` из `GET /camera/cooling/status` → `target_temp_c`.  
- Если setpoint в MaxIm «застыл», типичные причины: не обновлён зарегистрированный `ASCOM.ProjectR1.Camera.dll`, кэш клиента, либо цель менялась только в UI, а MaxIm не отправлял новый setpoint — по trace-логу драйвера это видно.

### 3.5. Порт 3037 и «ничего не работает»

- Ошибка `ERR_CONNECTION_REFUSED` означает, что **процесс сервиса на 127.0.0.1:3037 не запущен**. Нужно снова запустить `uvicorn` из каталога `02_Python_Camera_Service` (или ваш штатный скрипт/служба).

## 4. Проверка после обновления

1. `GET http://127.0.0.1:3037/api/v1/health` → `status: ok`.  
2. `GET http://127.0.0.1:3037/api/v1/camera/cooling/debug` → ожидается актуальный `runtime_version`.  
3. Автотесты: `pytest` в `02_Python_Camera_Service` (включая регрессии охлаждения и warm-up).  
4. После замены DLL — перерегистрация/копирование `ASCOM.ProjectR1.Camera.dll` согласно вашей процедуре установки драйвера.
