# Резюме по замечаниям заказчика (Замечания1)

Дата фиксации: 2026-03-09  
Источник: `Замечания1.txt`

## 1. Краткий итог встречи

- Заказчик подтвердил, что базовая цепочка уже рабочая: камера -> SDK -> сервис -> ASCOM -> MaxIm.
- Текущая веб-панель оценена положительно как тестовый/настроечный интерфейс.
- Получение кадров и базовая эксплуатация признаны рабочими.
- Основные запросы теперь смещаются в сторону удобства эксплуатации и параметризации под реальную обсерваторную работу.

## 2. Что важно оставить без изменений

- Логику сохранения профилей настроек в БД (и последующую загрузку) — это отмечено как сильная сторона.
- Текущий набор базовых параметров (скорость, gain-режим, охлаждение, затвор, сенсор) — нужен и полезен.
- Наличие warmup-механики на стороне сервиса — оставить как основу безопасного вывода камеры из охлаждения.

## 3. Ключевые замечания заказчика (по сути)

1) **UI/Setup-диалог ASCOM**
- Нужен формат, похожий на стандартный ASCOM `Properties`: модальное окно с основными режимами работы камеры.
- В перспективе: выбор камеры и порта в интерфейсе, поддержка 2+ камер.

2) **Readout speed**
- Сейчас ввод «произвольным числом» неудобен и не соответствует фактическим режимам SDK.
- Нужно заменить на фиксированный список допустимых скоростей из SDK/manual:
  - `50 kHz`, `100 kHz`, `250 kHz`, `500 kHz`, `1 MHz`, `3 MHz`.

3) **Gain**
- Сейчас используется техническое кодирование `0/1`, что непонятно оператору.
- Нужно показывать человекочитаемые режимы:
  - `High capacity mode` (большой динамический диапазон),
  - `Low noise mode` (низкий шум).
- Внутри можно хранить/передавать `0/1`, но в UI обязателен понятный label/tooltip.

4) **Cooler power**
- Сейчас значение мощности кулера выглядит как «застывшее» (около 50%), что не отражает реальную динамику.
- Требование: корректно отображать и передавать фактическую мощность кулера в ASCOM/клиент.
- Важно: пользователь **не просит ручную установку процента**; нужен корректный автоматический алгоритм регулирования.

5) **Эксплуатация охлаждения и warmup**
- Нужен безопасный, понятный сценарий «охлаждение/нагрев/отключение» при работе из астрософта.
- Отдельно поднят вопрос, как корректно дать функциональность warmup в контуре ASCOM-клиентов.

6) **Усилители (amplifier/readout channels)**
- Нужно исследовать режимы 1/2/4 усилителей, их влияние на скорость считывания и шум.
- По текущим наблюдениям, вероятно используется 1 усилитель по умолчанию.
- Решение должно опираться на реальные измерения/отчеты по камерам.

7) **Внешние материалы и экспертиза**
- Заказчик ожидает подключение внутренних отчетов по измерениям (Шмагин/Шугаров) и, при необходимости, консультаций по режимам эксплуатации.

## 4. Этап 2: обязательные задачи проекта

## 4.1 Обязательный функциональный объем (must-have)

1. Реализовать ASCOM `Properties/Setup`-диалог операторского уровня:
- выбор профиля камеры;
- выбор камеры/порта;
- readout speed как фиксированный combobox;
- gain как человекочитаемые режимы (`High capacity` / `Low noise`);
- сохранение/загрузка пресетов.

2. Перевести `readout_speed` на строго дискретные режимы SDK:
- убрать свободный ввод;
- добавить валидацию на backend;
- вернуть понятное сообщение при недопустимом значении.

3. Нормализовать отображение gain:
- UI labels + tooltip с физическим смыслом;
- документировать соответствие `label -> SDK int code`.

4. Завершить контур `cooler_power`:
- корректное чтение фактической мощности;
- стабильное отображение в API/ASCOM;
- исключить «залипание» значения.

5. Проработать алгоритм авто-регулирования охлаждения:
- логика удержания target без перерегулирования;
- учет внешней температуры/режима эксплуатации;
- защита от слишком агрессивного охлаждения.

6. Описать и внедрить сценарий warmup для оператора:
- четкий workflow перед disconnect/shutdown;
- совместимость с ASCOM-клиентами (где нет отдельной кнопки warmup).

## 4.2 Исследовательско-инженерный объем (R&D этапа 2)

7. Провести исследование режимов усилителей (1/2/4 канала):
- влияние на скорость и шум;
- рекомендации по режимам для спектральных задач;
- фиксация дефолтного режима в профилях.

8. Интегрировать результаты реальных тестов камеры:
- учесть внутренние отчеты и практику эксплуатации;
- при необходимости — отдельный документ «рекомендованные режимы».

9. Выполнить длительный hardware-soak с критериями приемки:
- стабильность экспозиций;
- стабильность cooling/warmup;
- отсутствие деградации в MaxIm/ASCOM-клиентах.

## 4.3 Запланированные исследования (детализация)

### A) Усилители (пункт 6 из замечаний)
- Цель: выбрать рекомендованный режим `1/2/4 amplifier` для разных задач (спектры, быстрые серии, калибровки).
- Что измеряем:
  - время считывания кадра;
  - read noise (e- RMS);
  - fixed pattern noise;
  - равномерность фона по зонам кадра;
  - устойчивость на сериях (не менее 200 кадров на режим).
- Матрица тестов:
  - температуры: `-5C`, `-10C`, `-15C`;
  - readout speed: `50kHz/250kHz/1MHz`;
  - gain mode: `high capacity` и `low noise`;
  - amplifiers: `1/2/4` (если доступны моделью).
- Артефакты:
  - таблица «шум/скорость/рекомендация»;
  - профиль по умолчанию для каждого сценария наблюдений.

### B) Алгоритм охлаждения (auto cooler control)
- Цель: автоматическое управление охлаждением без ручного процента мощности.
- Что проверяем:
  - скорость выхода на setpoint;
  - перерегулирование (overshoot);
  - время удержания в диапазоне `+/-0.3C`;
  - риск конденсации (запас к точке росы);
  - стабильность на ночных перепадах наружной температуры.
- Сценарии:
  - сухой холод;
  - влажная ночь;
  - быстрые перепады погоды;
  - restart сервиса в процессе сессии.
- Артефакты:
  - параметры регулятора по профилям;
  - режимы Auto: `Safe`, `Balanced`, `Fast`;
  - отчет soak-теста не менее 8 часов.
