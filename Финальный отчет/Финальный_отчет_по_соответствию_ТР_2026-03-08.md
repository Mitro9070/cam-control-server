# Финальный отчет по Project_R1 и соответствию ТР

Дата: 2026-03-08  
Базовый ТР: `техническое-решение-greateyes/01_ТР_GreatEyes_ASCOM_Python_PostgreSQL.md`

## 1. Что выполнено в рамках работ

### 1.1 Инфраструктура и развертывание
- Выполнен деплой Python сервиса и ASCOM слоя на текущем ПК.
- Настроен запуск API/UI на `127.0.0.1:3037`.
- Подключен `native` режим SDK (`greateyes.dll`) с сетевыми параметрами камеры.

### 1.2 Интеграция GreatEyes SDK
- Реализован сетевой bootstrap SDK:
  - `SetupCameraInterface`
  - `ConnectToSingleCameraServer` (с fallback)
  - `ConnectCamera`
  - `InitCamera`
- Добавлен `unlock/retry/backoff` и сериализация SDK-вызовов (lock) для снижения `camera busy`.
- Реализована и закреплена логика level-mode охлаждения (`target_temp_c -> level`) по бизнес-правилу из ТР.

### 1.3 API и доменная логика
- Реализованы сценарии:
  - connect/disconnect/state/capabilities
  - exposure start/status/stop/abort
  - latest image
  - cooling power/target/status/warmup
  - settings/logs
- Добавлено поле `percent` в `exposure status`.
- Добавлен совместимый endpoint warm-up статуса:
  - `GET /camera/cooling/warmup/{id}/status`
  - плюс сохранен `GET /camera/cooling/warmup/{id}`.
- Добавлен финальный wire-format для C# `ImageArray`:
  - `GET /camera/images/latest/raw` (`application/octet-stream`) + metadata headers.

### 1.4 База данных и миграции
- Расширена схема БД до R1.1 scope:
  - `camera_profiles`
  - `camera_session`
  - `exposure_job`
  - `exposure_image_meta`
- Добавлена миграция:
  - `02_Python_Camera_Service/alembic/versions/20260308_0002_operational_tables.py`
- Добавлена запись lifecycle данных для postgres режима (сессии, экспозиции, метаданные кадров).

### 1.5 Тесты
- Python: `pytest` проходит.
- UI: Playwright smoke проходит.
- C#: добавлен unit test проект `ASCOM.ProjectR1.Camera.Tests` и включен в solution.

## 2. Соответствие начальному ТР (оценка)

| Раздел ТР | Статус | Комментарий |
|---|---|---|
| Архитектура `ASCOM -> Python -> SDK` | Соответствует | Реализована, сервис/драйвер связаны через API. |
| Базовые API/операционные сценарии | Соответствует | Ключевые endpoints и сценарии реализованы. |
| Cooling + warm-up | Соответствует с ограничениями | Правило level-mode формализовано и реализовано; нужен длительный hardware-soak. |
| Wire-format для `ImageArray` | Соответствует | Добавлен raw endpoint и интеграция в C# c fallback. |
| R1.1 PostgreSQL операционные таблицы | Соответствует | Таблицы и миграция добавлены. |
| Автотесты (unit/integration/e2e) | Частично соответствует | Python/UI закрыты, C# unit базово добавлены; требуется дальнейшее расширение покрытия. |
| Нефункциональная стабильность native SDK | Частично соответствует | Есть hardening в коде; требуется длительная валидация на железе. |

## 3. Актуальный итог по GAP

- Закрыто:
  - as-built API контракт;
  - `percent` в статусе экспозиции;
  - warm-up `/status` endpoint;
  - финальный wire-format `ImageArray` (raw);
  - расширенная схема БД R1.1;
  - базовый C# unit test контур.
- Частично закрыто:
  - устойчивость native SDK в длительном режиме (`camera busy` требует soak-подтверждения);
  - финальный приемочный минимум как эксплуатационный критерий (длительный прогон).

## 4. Вывод

Проект приведен к состоянию высокой функциональной близости к начальному ТР.  
Ключевые архитектурные и контрактные гэпы закрыты в коде и документации.  
Оставшийся риск носит эксплуатационный характер (длительная стабильность native SDK на железе), а не архитектурно-контрактный.
