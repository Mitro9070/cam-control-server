# Отчет: нативные тесты с реальной камерой (полная диагностика)

Дата: 2026-03-07  
Контур: `ASCOM Local Server + Python camera-service`  
Режим SDK: `native`  
SDK DLL: `C:\Program Files\greateyes\greateyesVision\greateyes.dll`  
IP камеры: `192.168.31.234`

## 1) Что проверено в системе
- ASCOM Platform установлен, драйвер `GreatEyes 9.0 (Project_R1 ASCOM Camera)` зарегистрирован.
- Python service запущен в `.venv` и слушает `127.0.0.1:3037`.
- `GET /api/v1/health` -> `{"status":"ok","storage_mode":"postgres","db":"connected"}`.
- `http://127.0.0.1:3037/ui/` -> `200 OK`.
- Сетевой доступ до камеры есть: порт `12345` достижим (`TcpTestSucceeded=true`).

## 2) Проверка native API (без mock)

### 2.1 Соединение с камерой
- `POST /api/v1/camera/connect` -> **FAILED**  
  `SdkErrorCode.SDK_CALL_FAILED: ConnectCamera failed: no camera detected`
- `GET /api/v1/camera/state` -> `connected=false`, `camera_state=disconnected`.
- `GET /api/v1/camera/capabilities` -> **FAILED** (`NOT_CONNECTED`).

### 2.2 Экспозиции и охлаждение
- `PUT /camera/config/roi-binning` -> **FAILED** (`NOT_CONNECTED`).
- `POST /camera/exposures` -> **FAILED** (`NOT_CONNECTED`).
- `PUT /camera/cooling/power` и `PUT /camera/cooling/target` -> **FAILED** (`NOT_CONNECTED`).
- `GET /camera/cooling/status` возвращает fallback-состояние (без реального link к камере).

## 3) Полная диагностика greateyesVision/SDK

### 3.1 Установка и конфиги вендора
- Установлен `greateyesVision version 8` в `C:\Program Files\greateyes\greateyesVision\`.
- Найдены два `greateyes.ini`:
  - `C:\Program Files\greateyes\greateyesVision\greateyes.ini` (старый, `Camera0_Address=192.168.1.234`, `Camera0_Interface=0`).
  - `C:\ProgramData\greateyes\2\greateyes.ini` (актуальный, `Camera0_Address=192.168.31.234`, `Camera0_Interface=3`).

### 3.2 Проверка работы greateyesVision в рантайме
- Процесс `greateyesVision.exe` запущен.
- Зафиксировано активное TCP-соединение:  
  `192.168.31.231:<ephemeral> -> 192.168.31.234:12345 (ESTABLISHED)`.
- Это подтверждает: вендорское ПО работает с камерой по сети.

### 3.3 Проверка текущего Python native-адаптера
- Адаптер проекта использует только вызов `ConnectCamera(camera_index)` и не делает сетевую инициализацию камеры.
- При `camera/connect` у процесса Python **нет TCP-соединения к `192.168.31.234:12345`** (есть только БД и localhost).
- Низкоуровневый вызов `ConnectCamera` через `ctypes` (индексы `0..3`) стабильно возвращает статус `1` (`no camera detected`).

### 3.4 Экспорты SDK DLL
- `greateyes.dll` экспортирует не только базовые функции (`ConnectCamera`, `StartMeasurement`, ...), но и сетевые (`ConnectToCameraServer`, `ConnectToSingleCameraServer`, `SetupCameraInterface`, `SetConnectionType` и др.).
- Эти сетевые функции в текущем коде `Project_R1` не используются.

## 4) Технический вывод
- Блокер не в деплое и не в доступности камеры по IP.
- Блокер в несовпадении режима интеграции:
  - `greateyesVision` работает в сетевом режиме (порт `12345`),
  - текущий native-адаптер `Project_R1` использует только базовый путь `ConnectCamera` без явной сетевой инициализации.
- Статус: **BLOCKED (integration gap in SDK call sequence for network camera mode)**.

## 5) Что нужно для снятия блокера
- Подтвердить сигнатуры и порядок вызовов сетевых API из SDK (документация/headers от вендора).
- Добавить в `sdk_adapter.py` сетевой bootstrap перед `ConnectCamera`:
  - выбор интерфейса/типа соединения,
  - установка адреса/режима,
  - подключение к camera server (`192.168.31.234:12345`),
  - и только потом стандартный `ConnectCamera`.
- После внедрения повторно прогнать полный native smoke (`connect -> capabilities -> exposure -> cooling`).

## 6) Что доработано в коде и результат
- Добавлены новые параметры конфигурации сервиса:
  - `SDK_CAMERA_ADDRESS`
  - `SDK_CAMERA_INTERFACE`
  - `SDK_CAMERA_PORT`
- В `NativeGreatEyesSdkAdapter` добавлена синхронизация `greateyes.ini` перед `ConnectCamera`:
  - обновляются `Camera{index}_Address` и `Camera{index}_Interface`
  - целевые ini: рядом с DLL и `C:\ProgramData\greateyes\2\greateyes.ini`.
- В `.env` выставлены:
  - `SDK_CAMERA_ADDRESS=192.168.31.234`
  - `SDK_CAMERA_INTERFACE=3`
  - `SDK_CAMERA_PORT=12345`
- После перезапуска и повторного native smoke:
  - результат `camera/connect` остался `ConnectCamera failed: no camera detected`.

Итог по доработке: подготовка сетевого профиля в ini теперь автоматизирована, но без точной сигнатуры и последовательности сетевых вызовов SDK (`ConnectTo*`/`Setup*`) камера из Python все еще не поднимается.

## 9) Расширенный сигнатурный probe сетевых функций
- Добавлен скрипт:
  - `02_Python_Camera_Service/tools/probe_greateyes_network_handshake.py`
- Что делает:
  - запускает отдельные подпроцессы для разных гипотез сигнатур/последовательностей
  - тестирует `CDLL/WinDLL` и комбинации вызовов `ConnectTo*`, `Setup*`, `SetConnectionType`, `Init*`, `Check*`
  - после каждого сценария проверяет `ConnectCamera`.
- Результат:
  - ни один сценарий не дал `ConnectCamera status=0`;
  - повторяются статусы `1 (no camera detected)` и `10 (camera busy)`;
  - для ряда комбинаций есть access violation, что подтверждает несоответствие сигнатур.

## 10) Найденные точные прототипы и внедрение
- В `repos` найден заголовок SDK:
  - `repos/SimplestNativeDLL/Release/greateyes.h`
- Из него взяты и внедрены ключевые сигнатуры/порядок для Ethernet:
  - `SetupCameraInterface(int connectionType, char* ipAddress, int& statusMSG, int addr)`
  - `ConnectToSingleCameraServer(int addr)` (+ fallback `ConnectToMultiCameraServer()`)
  - `ConnectCamera(...)`
  - `InitCamera(int& statusMSG, int addr)`
- В `sdk_adapter.py` реализована последовательность:
  - `SetupCameraInterface -> ConnectToSingleCameraServer -> ConnectCamera -> InitCamera`
  - с очисткой stale session через `DisconnectCameraServer`.
